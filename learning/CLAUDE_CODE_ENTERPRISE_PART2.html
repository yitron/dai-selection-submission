4. Governance

Permissions, security policies, sandboxing, access control

Governance Architecture

⚠️ Security Model: Claude Code implements a multi-layered permission system with:
  • Permission modes (ask, auto, bypass)
  • Tool-level allow/deny rules
  • Dangerous command detection
  • Sandbox execution (Bubblewrap/Docker)
  • Organization policy limits (enterprise)
  • Remote managed settings

Permission System Layers

Permission Check Flow

Layer 1: Permission Mode Check
// Permission modes (stored in settings) type PermissionMode = 'ask' | 'auto' | 'bypass'; // ask: Always prompt user for approval (default) // auto: Auto-approve safe tools, prompt for dangerous (with classifier) // bypass: Never prompt (dangerous, testing only) const mode = getPermissionMode(); // From settings.json
⬇
Layer 2: Policy Limits Check (Enterprise)
// services/policyLimits/index.ts // Organization-level restrictions from API const isPolicyAllowed = async (feature: string): Promise => { const limits = await getPolicyLimits(); // Example restrictions: // - allow_remote_control: false // - allow_web_search: false // - max_token_usage: 1000000 // - allowed_models: ['claude-sonnet-4'] return !limits.restrictions[feature]; }; // Fails OPEN on errors (non-blocking) // Cached with ETag for performance // Background polling every hour
⬇
Layer 3: Tool Permission Rules
// utils/permissions/PermissionRule.ts type PermissionRule = { tool: string; // Tool name (e.g., 'Bash', 'FileEdit') content?: string; // Optional content matcher value: 'allow' | 'deny'; // Rule action source: PermissionRuleSource; // Where rule came from }; // Examples: // Allow all git commands: { tool: 'Bash', content: 'git*', value: 'allow' } // Deny destructive git operations: { tool: 'Bash', content: 'git push --force*', value: 'deny' } // Allow reading any file: { tool: 'FileRead', value: 'allow' } // Deny editing .env files: { tool: 'FileEdit', content: '**/.env', value: 'deny' }
⬇
Layer 4: Dangerous Pattern Detection
// utils/permissions/dangerousPatterns.ts const DANGEROUS_BASH_PATTERNS = [ 'python', 'python3', 'node', 'ruby', 'perl', 'curl', 'wget', 'nc', 'netcat', 'rm -rf', 'chmod', 'chown', 'eval', 'exec', 'sudo', ]; // For auto mode: block dangerous patterns even with allow rules const isDangerous = isDangerousBashPermission(toolName, command); if (mode === 'auto' && isDangerous) { // Always prompt, never auto-approve return requireUserApproval(); }
⬇
Layer 5: Sandbox Execution
// utils/sandbox/sandbox-adapter.ts class SandboxManager { static isSandboxingEnabled(): boolean { // Check if Bubblewrap (Linux) or Docker available return hasBuiltinSandbox() || hasDockerSandbox(); } static executeInSandbox(command: string): Promise { // Create isolated environment: // - Read-only /usr, /bin, /lib // - Writable workspace directory only // - No network (unless explicitly allowed) // - Limited PIDs, memory, CPU return bubblewrap.run({ command, allowedPaths: [workspaceDir], network: false, timeout: 120000, }); } }

5. Multi-Agent Orchestration

Coordinator mode, agent spawning, team management, inter-agent communication

Multi-Agent Architecture

✅ Multi-Agent Patterns: Claude Code supports TWO multi-agent models:
  • Coordinator Mode: Single coordinator agent + multiple worker agents
  • Team Mode: Peer-to-peer agent collaboration with shared memory

Coordinator Mode Architecture

coordinator/coordinatorMode.ts src/coordinator/coordinatorMode.ts
Coordinator mode implementation - orchestrate multiple worker agents
Coordinator Capabilities:
  • Spawn worker agents via AgentTool
  • Send follow-up messages via SendMessageTool
  • Stop workers via TaskStopTool
  • Receive worker results as user-role messages (XML format)
  • Subscribe to GitHub PR events
Worker Restrictions:
  • Limited tool access (Bash, Read, Edit, MCP tools)
  • Cannot spawn sub-agents (flat hierarchy)
  • Results sent back to coordinator automatically
  • Isolated conversation context

Key Architecture Patterns to Replicate

  1. Bootstrap State Pattern: Centralized session metadata with zero circular dependencies
  2. Tool System: Modular, permission-aware tools with common interface
  3. Multi-Layer Governance: Permission modes → Policy limits → Tool rules → Dangerous patterns → Sandbox
  4. Span-Based Tracing: Hierarchical tracing (interaction → llm_request → tool) for debugging
  5. Event-Driven Telemetry: logEvent() sink pattern with PII protection
  6. Session Recovery: JSONL-based conversation logs for resume/debugging
  7. Agent Isolation: Per-agent memory, conversation context, color assignment
  8. Fail-Open Policies: API errors don't block execution (graceful degradation)

Critical Differences for Multi-User

⚠️ Don't Just Scale the CLI:
  • Replace global STATE with per-request context (user ID, org ID, session ID)
  • Add authentication layer (OAuth, SAML, API keys)
  • Implement resource quotas (API calls, agents, storage per user/org)
  • Add billing/metering (track costs per user/org)
  • Implement rate limiting at user/org level
  • Add cross-user access controls (prevent session hijacking)
  • Implement audit logging for compliance
  • Add admin UI for org management

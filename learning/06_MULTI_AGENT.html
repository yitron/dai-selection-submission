🎯 1. Coordinator Mode (Hierarchical)

coordinator/coordinatorMode.ts src/coordinator/coordinatorMode.ts
Coordinator mode implementation - orchestrate multiple worker agents
Coordinator Capabilities:
  • Spawn worker agents via Task tool
  • Send follow-up messages via SendMessage tool
  • Stop workers via TaskStop tool
  • Receive worker results as user-role messages (XML format)
  • Subscribe to GitHub PR events
Worker Restrictions:
  • Limited tool access (Bash, Read, Edit, MCP tools)
  • Cannot spawn sub-agents (flat hierarchy)
  • Results sent back to coordinator automatically
  • Isolated conversation context

Coordinator Message Flow

1. User Request
User: "Implement authentication and write tests"
⬇
2. Coordinator Planning
Coordinator (to user): "I'll spawn two workers: 1. Worker A: Implement authentication logic 2. Worker B: Write comprehensive tests" // Spawns agents await Task({ prompt: "Implement OAuth2 authentication in src/auth.ts", subagent_type: "general-purpose" }); // Returns agent_id: "agent_abc123" await Task({ prompt: "Write tests for authentication in tests/auth.test.ts", subagent_type: "general-purpose" }); // Returns agent_id: "agent_def456"
⬇
3. Workers Execute (Parallel)
// Worker A executes independently Worker A: - Reads existing code - Implements OAuth2 logic - Uses Bash, Edit tools - Returns result to coordinator // Worker B executes in parallel Worker B: - Reads authentication code - Writes test cases - Runs tests via Bash - Returns result to coordinator
⬇
4. Results Arrive as User Messages (XML)
// Worker results arrive as XML in user-role messages <task-notification> <task-id>agent_abc123</task-id> <status>completed</status> <result> I've implemented OAuth2 authentication in src/auth.ts with: - Login endpoint with JWT tokens - Refresh token flow - Logout endpoint </result> <usage> <total_tokens>12543</total_tokens> <tool_uses>8</tool_uses> </usage> </task-notification> // Coordinator sees this as a USER message (not assistant) // This is how the coordinator receives worker outputs
⬇
5. Coordinator Synthesis
Coordinator (to user): "Both workers completed successfully: ✅ Authentication implemented (Worker A) ✅ Tests written and passing (Worker B) The OAuth2 flow is ready with 15 passing tests."
Key Pattern: Worker results arrive as user-role messages with XML content. This allows the coordinator to process multiple worker outputs in a single turn.

👥 2. Team Mode (Peer-to-Peer)

Team Mode vs Coordinator Mode:
  • Coordinator: Hierarchical (1 leader + N workers)
  • Team: Flat (N peers collaborating)
utils/swarm/ src/utils/swarm/ (15+ files)
Team mode implementation - peer agent collaboration
Key Concepts:
  • Teammates: Agents in the same team (shared team name)
  • Shared Memory: Team-scoped MEMORY.md file
  • Cross-Agent Messages: Send messages between teammates
  • Team Lifecycle: TeamCreate, TeamDelete tools
Key Files:
  • teamHelpers.ts: Team creation/deletion
  • teammatePromptAddendum.ts: Team context injection
  • teammateModeSnapshot.ts: Team state snapshots
  • reconnection.ts: Reconnect to existing teams

Team Mode Example

// Create a team await TeamCreate({ teamName: "auth-dev-team" }); // Spawn teammate agents await Task({ prompt: "Implement backend authentication", team: "auth-dev-team" }); await Task({ prompt: "Implement frontend login UI", team: "auth-dev-team" }); // Agents share: // - Team-scoped MEMORY.md // - Can send messages to each other // - Collaborate on shared codebase

🔧 3. Standard Agents (Parent-Child)

Standard agent spawning system
Key Files:
  • AgentTool.tsx (234KB): Main agent tool implementation
  • runAgent.ts (36KB): Agent execution loop
  • forkSubagent.ts: Spawn child process for agent
  • resumeAgent.ts: Resume paused agents
  • agentMemory.ts: Agent-specific memory isolation
  • loadAgentsDir.ts (26KB): Load custom agent definitions
  • built-in/: Built-in agent types (Explore, Plan, Bash, etc.)

Agent Types

Agent Type Purpose Available Tools
general-purpose Full-featured agent for complex tasks All tools (Bash, File ops, Web, MCP, sub-agents)
Explore Fast codebase exploration Glob, Grep, Read (no Edit/Write)
Plan Implementation planning Glob, Grep, Read, ExitPlanMode
Bash Command execution specialist Bash only
test-runner Run tests after code changes Bash, Read
Custom Agents User-defined agent types Configurable via .agents/ directory

🌐 4. KAIROS (Remote Session Access)

KAIROS (Assistant Mode): Connect to remote CCR (Claude Code Remote) sessions running on claude.ai
// Connect to remote session $ claude assistant [session-id] // Features: // - View-only mode (no sending commands directly) // - See session activity in real-time // - Access via WebSocket to claude.ai API // - Useful for customer support / debugging
Use Case: Enterprise support teams can view user sessions to help debug issues without taking over control.

🏢 Enterprise Multi-Agent Adaptations

Multi-Tenant Agent System:

For enterprise systems, enhance multi-agent capabilities:

1. Agent Pools

  • Pre-warmed agent instances per user/org
  • Kubernetes pods with auto-scaling
  • Agent instance reuse to reduce cold starts
// Enterprise agent pool class AgentPool { private pools: Map = new Map(); async getAgent(orgId: string): Promise { const pool = this.pools.get(orgId) || []; // Try to reuse warm agent if (pool.length > 0) { return pool.pop()!; } // Spawn new agent pod return await spawnAgentPod({ orgId, resources: { cpu: '1 core', memory: '2GB' } }); } async releaseAgent(orgId: string, agent: Agent): void { const pool = this.pools.get(orgId) || []; pool.push(agent); this.pools.set(orgId, pool); } }

2. Agent Quotas

  • Limit concurrent agents per user (e.g., max 5)
  • Different quotas for free/pro/enterprise tiers
  • Queue agent requests when limit reached

3. Agent Scheduling

  • Priority queues for premium users
  • Fair scheduling across users
  • Agent preemption for critical tasks

4. Cross-User Agents

  • Share agents across org (with permissions)
  • Org-wide agent library
  • Agent marketplace for custom agents

5. Agent Monitoring

  • Real-time dashboard of active agents
  • Agent health checks & auto-recovery
  • Cost tracking per agent

6. Agent Billing

  • Track agent usage costs per user/org
  • Charge based on agent-hours or tokens
  • Budget alerts and auto-stop

7. Hierarchical Teams

  • Multi-level coordinator hierarchies
  • Coordinator of coordinators pattern
  • Scale to 100+ agents per task

🔐 Permission Check Flow (5 Layers)

Defense-in-Depth Security

Layer 1: Permission Mode Check
// Permission modes (stored in settings) type PermissionMode = 'ask' | 'auto' | 'bypass'; // ask: Always prompt user for approval (default) // auto: Auto-approve safe tools, prompt for dangerous (with classifier) // bypass: Never prompt (dangerous, testing only) const mode = getPermissionMode(); // From settings.json
⬇
Layer 2: Policy Limits Check (Enterprise)
// services/policyLimits/index.ts // Organization-level restrictions from API const isPolicyAllowed = async (feature: string): Promise => { const limits = await getPolicyLimits(); // Example restrictions: // - allow_remote_control: false // - allow_web_search: false // - max_token_usage: 1000000 // - allowed_models: ['claude-sonnet-4'] return !limits.restrictions[feature]; }; // Fails OPEN on errors (non-blocking) // Cached with ETag for performance // Background polling every hour
⬇
Layer 3: Tool Permission Rules
// utils/permissions/PermissionRule.ts type PermissionRule = { tool: string; // Tool name (e.g., 'Bash', 'FileEdit') content?: string; // Optional content matcher value: 'allow' | 'deny'; // Rule action source: PermissionRuleSource; // Where rule came from }; // Examples: // Allow all git commands: { tool: 'Bash', content: 'git*', value: 'allow' } // Deny destructive git operations: { tool: 'Bash', content: 'git push --force*', value: 'deny' } // Allow reading any file: { tool: 'Read', value: 'allow' } // Deny editing .env files: { tool: 'Edit', content: '**/.env', value: 'deny' }
⬇
Layer 4: Dangerous Pattern Detection
// utils/permissions/dangerousPatterns.ts const DANGEROUS_BASH_PATTERNS = [ 'python', 'python3', 'node', 'ruby', 'perl', 'curl', 'wget', 'nc', 'netcat', 'rm -rf', 'chmod', 'chown', 'eval', 'exec', 'sudo', ]; // For auto mode: block dangerous patterns even with allow rules const isDangerous = isDangerousBashPermission(toolName, command); if (mode === 'auto' && isDangerous) { // Always prompt, never auto-approve return requireUserApproval(); }
⬇
Layer 5: Sandbox Execution
// utils/sandbox/sandbox-adapter.ts class SandboxManager { static isSandboxingEnabled(): boolean { // Check if Bubblewrap (Linux) or Docker available return hasBuiltinSandbox() || hasDockerSandbox(); } static executeInSandbox(command: string): Promise { // Create isolated environment: // - Read-only /usr, /bin, /lib // - Writable workspace directory only // - No network (unless explicitly allowed) // - Limited PIDs, memory, CPU return bubblewrap.run({ command, allowedPaths: [workspaceDir], network: false, timeout: 120000, }); } }

🤖 Auto Mode Classifier (Advanced)

Auto Mode: Uses a separate LLM call to classify tool requests as safe/unsafe before auto-approving.
// utils/permissions/autoModeState.ts // Classifier prompt: "Is this Bash command safe to auto-execute?" const classifierPrompt = ` You are a security classifier. Determine if this tool use is safe to auto-approve. Tool: ${toolName} Input: ${JSON.stringify(toolInput)} Rules: - Auto-approve: Read operations, safe git commands, npm install - Require approval: Write operations, rm/destructive commands, network access Respond with: SAFE or UNSAFE `; const classification = await llm.classify(classifierPrompt); if (classification === 'SAFE') { // Auto-approve return { approved: true, source: 'auto-classifier' }; } else { // Prompt user return await showApprovalDialog(toolName, toolInput); }
Cost Consideration: Auto mode classifier makes an extra LLM call per tool use. Can be expensive for high-frequency tool usage.

🏢 Enterprise Governance Enhancements

Multi-Tenant Governance:

For enterprise systems, you'll need enhanced governance:

1. RBAC (Role-Based Access Control)

  • Roles: Admin, Developer, Analyst, Viewer
  • Permissions per role: Different tool access levels
  • Role inheritance: Admin inherits Developer permissions
// Enterprise RBAC const ROLE_PERMISSIONS = { admin: ['*'], // All tools developer: ['Bash', 'Read', 'Write', 'Edit', 'Glob', 'Grep'], analyst: ['Read', 'Glob', 'Grep', 'WebSearch'], viewer: ['Read', 'Glob'] }; function hasToolPermission(userId: string, toolName: string): boolean { const user = getUser(userId); const allowedTools = ROLE_PERMISSIONS[user.role]; return allowedTools.includes('*') || allowedTools.includes(toolName); }

2. Org Hierarchies

  • Parent org policies inherited by child orgs
  • Child orgs can be more restrictive (not less)
  • Policy conflict resolution

3. Audit Logging

  • Log all permission decisions with user/org context
  • Track permission denials for security monitoring
  • Export audit logs for compliance (SOC2, ISO27001)
// Audit log entry { timestamp: '2026-04-03T10:30:00Z', userId: 'user_abc123', orgId: 'org_xyz789', sessionId: 'ses_def456', decision: 'DENIED', tool: 'Bash', input: { command: 'rm -rf /' }, reason: 'dangerous_pattern_detected', layer: 4, // Which layer blocked it }

4. Custom Policies

  • Per-org dangerous pattern lists
  • Industry-specific restrictions (healthcare, finance)
  • Data residency requirements

5. Compliance Reporting

  • Export policy violations for SOC2/ISO27001
  • Generate compliance reports (monthly, quarterly)
  • Alert on high-risk activities

6. Break-Glass Access

  • Emergency bypass with logging & approval
  • Temporary elevated permissions
  • Post-incident review workflow

7. Time-Based Policies

  • Restrict certain features to business hours
  • Weekend/holiday lockdowns for sensitive operations
  • Maintenance windows with elevated permissions

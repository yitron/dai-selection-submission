🏗️ Tool System Architecture

Design Pattern: Claude Code implements tools as independent modules with a common interface. Each tool is responsible for:
  • Input validation (Zod schemas)
  • Permission checks
  • Execution logic
  • Result formatting
  • Error handling

Tool Interface Definition

// src/Tool.ts - Core tool type definition export type Tool = { name: string; description: string; input_schema: ToolInputJSONSchema; // Execution execute(input: unknown, context: ToolContext): Promise; // Permission checking requiresApproval?: (input: unknown) => boolean; getApprovalMessage?: (input: unknown) => string; // Progress reporting onProgress?: (data: ToolProgressData) => void; }; export type ToolResult = { content: ContentBlock[]; isError?: boolean; metadata?: Record; }; export type ToolContext = { sessionId: string; cwd: string; abortSignal?: AbortSignal; };

⚙️ Tool Execution Pipeline

End-to-End Tool Execution

1. Tool Discovery (API Response)
// QueryEngine receives tool_use block from API { type: 'tool_use', id: 'toolu_01ABC', name: 'Bash', input: { command: 'ls -la', description: 'List files' } }
⬇
2. Tool Lookup
// tools.ts - getTools() registry const tool = getTools().find(t => t.name === 'Bash'); if (!tool) throw new Error('Tool not found');
⬇
3. Permission Check
// hooks/toolPermission/useCanUseTool.tsx const permitted = await useCanUseTool(tool, input); // Shows approval dialog if needed // Checks bypass permissions mode // Validates against policy limits
⬇
4. Execution
// Tool.execute() with telemetry const startTime = Date.now(); const span = startToolSpan(tool.name, input); try { const result = await tool.execute(input, context); logEvent('tengu_tool_use_completed', { toolName: sanitizeToolNameForAnalytics(tool.name), duration_ms: Date.now() - startTime, is_error: false, }); return result; } catch (error) { logEvent('tengu_tool_use_failed', { ... }); throw error; } finally { endToolSpan(span); }
⬇
5. Result Assembly
// Create tool_result message block const toolResult = { type: 'tool_result', tool_use_id: 'toolu_01ABC', content: result.content, is_error: result.isError, }; // Add to conversation and continue loop messages.push(toolResult); return query(messages); // Recursive call

📊 Tool Registry System

// src/tools.ts - Dynamic tool assembly export function getTools(): Tool[] { const tools: Tool[] = []; // Core tools (always available) tools.push(ReadTool); tools.push(WriteTool); tools.push(EditTool); tools.push(GlobTool); tools.push(GrepTool); // Conditional tools based on platform if (process.platform === 'win32') { tools.push(PowerShellTool); } else { tools.push(BashTool); } // Feature-gated tools if (getSettings().features.webSearch) { tools.push(WebSearchTool); } if (getSettings().features.agentSpawning) { tools.push(AgentTool); } // MCP tools (dynamically loaded) const mcpTools = getMCPTools(); tools.push(...mcpTools); // Permission filtering return tools.filter(tool => hasPermission(tool, getPermissionMode()) ); }

🏢 Enterprise Toolkit Adaptations

Multi-User Tool System:

For enterprise systems, you'll need to enhance the toolkit:

1. Tool Isolation

  • Problem: Tools share execution environment
  • Solution: Run tools in user-specific containers/sandboxes
// Enterprise: Per-user tool execution const result = await executeInUserContainer({ userId: context.userId, orgId: context.orgId, tool: 'Bash', command: input.command, resourceLimits: { cpu: '1 core', memory: '512MB', timeout: '5min', } });

2. Resource Limits

  • CPU/memory/time limits per user/org
  • Prevent resource exhaustion attacks
  • Fair resource allocation across users

3. Tool Quotas

  • Track tool usage per user (e.g., max 100 Bash calls/hour)
  • Different quota tiers (free, pro, enterprise)
  • Quota reset policies (hourly, daily, monthly)

4. Tool Allowlists

  • Per-org tool restrictions (disable BashTool for some orgs)
  • Role-based tool access (admin vs developer vs analyst)
  • Dynamic tool enablement based on plan/tier

5. Audit Logging

  • Log all tool executions with user/org context
  • Compliance requirements (SOC2, GDPR, HIPAA)
  • Forensic analysis for security incidents
// Enterprise audit log { timestamp: '2026-04-03T10:30:00Z', userId: 'user_abc123', orgId: 'org_xyz789', sessionId: 'ses_def456', tool: 'Bash', input: { command: 'git commit -m "Fix bug"' }, output: { exitCode: 0, stdout: '...' }, duration_ms: 1500, resourceUsage: { cpu: '0.5s', memory: '45MB' }, }

6. Tool Result Caching

  • Cache deterministic tool results (e.g., file reads)
  • Per-user cache isolation
  • Cache invalidation on file modifications
  • Cost savings on repeated operations

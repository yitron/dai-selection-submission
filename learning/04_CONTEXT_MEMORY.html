📜 File History Tracking

utils/fileHistory.ts src/utils/fileHistory.ts
Tracks recently accessed files for context injection
Mechanism:
  • Maintains LRU (Least Recently Used) cache of file paths
  • Automatically adds to system context when file count < threshold
  • Used for prompt caching optimization
  • Persisted per session
  • Helps LLM understand which files are relevant
// File history tracking class FileHistory { private recentFiles: string[] = []; private maxSize = 20; addFile(path: string) { // Remove if already exists (move to front) this.recentFiles = this.recentFiles.filter(f => f !== path); // Add to front this.recentFiles.unshift(path); // Trim to max size if (this.recentFiles.length > this.maxSize) { this.recentFiles = this.recentFiles.slice(0, this.maxSize); } } getRecentFiles(): string[] { return this.recentFiles; } }

♻️ Message Compaction

Why Compaction? Long conversations exceed context windows. Compaction summarizes old messages to stay within limits.

Compaction Strategies

// services/compact/compactMessages.ts // Strategy 1: Keep Recent + Summarize Middle async function compactMessages(messages: Message[]): Promise { // Always keep: const keepFirst = messages[0]; // First user message const keepLast = messages.slice(-10); // Last 10 messages // Summarize middle const middleMessages = messages.slice(1, -10); const summary = await summarizeMessages(middleMessages); return [ keepFirst, createAssistantMessage({ content: summary }), ...keepLast ]; } // Strategy 2: Remove Tool Results (keep tool_use, remove results) // Strategy 3: Combine consecutive assistant messages // Strategy 4: Remove thinking blocks (if using extended thinking)

🏢 Enterprise Memory Adaptations

Multi-User Memory System:

For enterprise systems, memory must be isolated and scaled:

1. User-Scoped Memory

  • Problem: MEMORY.md is per-project, not per-user
  • Solution: Store memories with user_id + project_id scope
// Enterprise memory storage memories/ user_abc123/ project_xyz789/ MEMORY.md project_def456/ MEMORY.md

2. Org-Wide Knowledge Base

  • Shared memory pool for entire organization
  • Company-wide coding standards, architecture patterns
  • Common troubleshooting knowledge

3. Vector Database for Memory

  • Problem: File-based memory doesn't scale to thousands of users
  • Solution: Use pgvector, Pinecone, or Weaviate
// Vector database schema CREATE TABLE memories ( id UUID PRIMARY KEY, user_id VARCHAR(255) NOT NULL, org_id VARCHAR(255) NOT NULL, project_id VARCHAR(255), content TEXT NOT NULL, embedding VECTOR(1536), -- OpenAI ada-002 memory_type VARCHAR(50), -- 'automatic' | 'manual' | 'summary' created_at TIMESTAMP, INDEX idx_user_org (user_id, org_id), INDEX idx_embedding USING ivfflat (embedding vector_cosine_ops) ); -- Semantic search SELECT content, 1 - (embedding <=> query_embedding) AS similarity FROM memories WHERE user_id = ? AND org_id = ? ORDER BY similarity DESC LIMIT 10;

4. Memory Access Control

  • Hierarchical memory (personal → team → org)
  • Role-based memory access (admin can see all org memories)
  • Memory sharing permissions

5. Memory Versioning

  • Track all memory changes for audit
  • Rollback capability for corrupted memories
  • Memory edit history

6. Cross-Session Memory

  • Persist user memories across all sessions
  • Automatically recall relevant memories from past projects
  • Memory retention policies (expire after N days)

📡 OpenTelemetry Integration

Three Observability Signals:
  • Metrics: Counters and gauges (session_count, cost, tokens)
  • Traces: Distributed tracing spans (interaction → llm → tool)
  • Logs: Structured event logs (tengu_* events)

Telemetry Architecture

Signal Type Technology Use Case Exporters
Metrics OpenTelemetry Metrics Counters (session_count, cost, tokens), Gauges (active_sessions) OTLP, Datadog, BigQuery
Traces OpenTelemetry Traces Span hierarchy (interaction → llm_request → tool) OTLP, Datadog
Logs OpenTelemetry Logs Event logs (tengu_* events) with structured metadata OTLP, BigQuery

🔍 Span Hierarchy

Critical Pattern: Claude Code uses a hierarchical span structure for distributed tracing.
// Span hierarchy interaction (root span) ├── llm_request │ ├── tool.execution (BashTool) │ │ └── tool.blocked_on_user (permission approval) │ └── tool.execution (FileReadTool) └── hook (user_prompt_submit_hook) // Example attributes on interaction span: { 'span.type': 'interaction', 'user.prompt': 'Fix the authentication bug', 'session.id': 'ses_abc123', 'session.model': 'claude-sonnet-4-5', 'duration_ms': 12450, }

Span Creation API

// utils/telemetry/sessionTracing.ts // 1. Start interaction span (user turn) export function startInteractionSpan(userPrompt: string): Span { const span = tracer.startSpan('interaction', { attributes: { 'span.type': 'interaction', 'user.prompt': truncateContent(userPrompt, 1000), 'session.id': getSessionId(), 'session.model': getMainLoopModel(), } }); interactionContext.enterWith({ span, startTime: Date.now() }); return span; } // 2. Start LLM request span (child of interaction) export function startLLMSpan(messages: Message[]): Span { return tracer.startSpan('llm_request', { attributes: { 'span.type': 'llm_request', 'message_count': messages.length, 'model': getMainLoopModel(), } }); } // 3. Start tool execution span (child of llm_request) export function startToolSpan(toolName: string, input: unknown): Span { return tracer.startSpan('tool.execution', { attributes: { 'span.type': 'tool', 'tool.name': toolName, 'tool.input': JSON.stringify(input), } }); }

🚨 Incident Control Capabilities

Critical for Enterprise: Session control enables real-time incident response, debugging, and user support.

1. Error Logging

// bootstrap/state.ts - In-memory error log export function addToInMemoryErrorLog(error: Error): void { STATE.inMemoryErrorLog.push({ error: errorMessage(error), timestamp: new Date().toISOString(), stack: error.stack, }); // Limited to last 100 errors if (STATE.inMemoryErrorLog.length > 100) { STATE.inMemoryErrorLog.shift(); } } // Retrieve for incident reports export function getInMemoryErrorLog(): ErrorLogEntry[] { return STATE.inMemoryErrorLog; }

2. Slow Operations Tracking

// bootstrap/state.ts - Track slow operations export function addSlowOperation(op: string, durationMs: number): void { if (durationMs > 1000) { // Only log operations > 1s STATE.slowOperations.push({ operation: op, durationMs: durationMs, timestamp: Date.now(), }); } } // Used for performance debugging // Example: "FileRead took 5432ms for large.csv"

3. Last API Request Capture

// bootstrap/state.ts export function setLastAPIRequest( request: BetaMessageStreamParams ): void { STATE.lastAPIRequest = omit(request, 'messages'); STATE.lastAPIRequestMessages = request.messages; } // Used for bug reports - exact request sent to API // Helps reproduce API errors

🎓 Model Training Data Pipeline

Training Data Collection: Session JSONL files are ideal for model fine-tuning and evaluation.

Training Pipeline Architecture

// Enterprise training data pipeline // 1. Session Collection // ~/.claude/sessions/*.jsonl → S3/GCS bucket // 2. Data Cleaning // - Remove PII (user paths, emails, secrets) // - Filter low-quality sessions (errors, incomplete) // - Normalize timestamps and IDs // 3. Format Conversion // JSONL → Fine-tuning format (Claude API format) [ { "role": "user", "content": "..." }, { "role": "assistant", "content": "..." } ] // 4. Quality Filtering // - High tool success rate // - Complete task resolution // - Positive user feedback (if available) // 5. Upload to Training Service // API call to Claude fine-tuning service // Or export for evaluation/analysis

Session Quality Metrics

Metric Description Use Case
Tool Success Rate % of tools that didn't error Filter high-quality sessions
Task Completion Session ended with natural stop Identify successful interactions
User Satisfaction Thumbs up/down (if collected) RLHF training data
Session Length Number of turns Filter too short/long sessions
Cost Efficiency Cost per successful task Optimize model performance

🏢 Enterprise Telemetry Adaptations

Multi-Tenant Telemetry:

For enterprise systems, enhance telemetry with multi-tenant labels:

1. Multi-Tenant Labels

// Add user/org labels to all telemetry const span = tracer.startSpan('interaction', { attributes: { 'user.id': userId, 'org.id': orgId, 'tenant.tier': 'enterprise', // free/pro/enterprise 'session.id': sessionId, } }); // Metrics with labels sessionCounter.add(1, { 'user.id': userId, 'org.id': orgId, 'model': 'claude-sonnet-4-5', });

2. Distributed Storage

  • Store session JSONL in S3/GCS (not local disk)
  • Partition by user_id and date
  • Lifecycle policies (archive after 90 days)

3. Real-Time Analytics

  • Stream telemetry to Kafka/Kinesis
  • Real-time dashboards (active users, costs)
  • Alert on anomalies (cost spikes, errors)

4. Cost Attribution

  • Track API costs per user/org
  • Chargeback to business units
  • Budget alerts and quota enforcement

5. Compliance & Audit

  • Tamper-proof audit logs
  • Data retention policies (GDPR, HIPAA)
  • Export for compliance reports

6. PII Protection

  • Redact PII from telemetry (paths, emails, keys)
  • Hash user IDs for analytics
  • Separate PII storage with encryption

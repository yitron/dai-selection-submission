🏗️ Claude Code Connection Patterns

✅ Three Connection Patterns:
Pattern Transport Use Case Latency
Bridge Mode HTTP long-polling IDE extensions (VS Code, Cursor) ~2s polling interval
Remote Sessions WebSocket (receive) + HTTP (send) Cloud CCR sessions from local CLI ~100ms (real-time)
Direct Connect WebSocket (bidirectional) Local server connections ~10ms (local network)

🌉 1. Bridge Mode (HTTP Polling)

Path: src/bridge/bridgeMain.ts
Use Case: IDE extensions connecting local machine to claude.ai

Bridge Architecture

1. Start Bridge Mode
$ claude remote-control # or $ claude bridge // Starts HTTP polling loop
⬇
2. Poll for Work (Every 2s)
// bridge/bridgeMain.ts async function pollForWork(api: BridgeApiClient, envId: string) { while (true) { // HTTP GET request to claude.ai API const work = await api.getWork(envId); if (work) { // Spawn child process for this session const handle = spawner.spawn({ sessionId: work.sessionId, workSecret: work.workSecret, cwd: work.cwd, }); } await sleep(2000); // Poll every 2 seconds } }
⬇
3. Execute Work (Child Process)
// Spawned child process runs full Claude Code // - Executes tools on local machine // - Sends results back to claude.ai via HTTP POST

Bridge Mode Characteristics

Aspect Implementation Trade-off
Latency 2-second polling interval Higher latency, but firewall-friendly
Firewall Outbound HTTP only (no inbound) Works behind corporate firewalls
Connection Client-initiated polling No persistent connection needed
Scalability One connection per environment Low overhead per client
// Bridge API endpoints POST /v1/bridge/environments/{envId}/poll → Returns work assignment or null POST /v1/bridge/sessions/{sessionId}/results → Sends tool execution results back

🔌 2. Remote Sessions (WebSocket + HTTP Hybrid)

Path: src/remote/SessionsWebSocket.ts
Use Case: Access cloud CCR sessions from local CLI

Remote Session Architecture

1. Connect to Remote Session
$ claude assistant [session-id] // Opens WebSocket connection to claude.ai
⬇
2. WebSocket Connection (Receive Only)
// remote/SessionsWebSocket.ts export class SessionsWebSocket { async connect(): Promise { const url = `${baseUrl}/v1/sessions/ws/${sessionId}/subscribe`; const ws = new WebSocket(url, { headers: { Authorization: `Bearer ${accessToken}` } }); ws.on('message', (data) => { const message = JSON.parse(data); if (message.type === 'control_request') { this.handlePermissionRequest(message); } else { this.callbacks.onMessage(message); } }); } }
⬇
3. HTTP POST for Sending (Not WebSocket)
// Sending messages uses HTTP POST (not WebSocket!) async sendMessage(message: string): Promise { await fetch(`${baseUrl}/v1/sessions/${sessionId}/messages`, { method: 'POST', headers: { Authorization: `Bearer ${accessToken}` }, body: JSON.stringify({ content: message }) }); } // Why HTTP for sending? // - Simpler retry logic // - Better error handling // - WebSocket used only for real-time receive
Hybrid Pattern: WebSocket for receiving (real-time updates) + HTTP POST for sending (reliability)

🖥️ 3. Direct Connect (Local WebSocket)

Command: claude open <url>
Use Case: Connect to local Direct Connect servers

Direct Connect Flow

// Start Direct Connect server (user's app) $ direct-connect-server --port 8080 // Connect from Claude Code CLI $ claude open ws://localhost:8080 // Establishes bidirectional WebSocket connection // - Send user messages via WebSocket // - Receive assistant responses via WebSocket // - Low latency (local network)
✅ Use Cases:
  • Local development servers
  • Custom integrations
  • Testing & debugging
  • Offline mode (no cloud required)

Full Name: Assistant Mode (KAIROS)

// Connect to remote CCR session $ claude assistant [session-id] // Features: // - View-only mode (cannot send commands directly) // - Real-time session activity // - WebSocket connection to claude.ai // - Used for customer support / debugging

🆚 Claude Code vs OpenClaw Gateway

Aspect Claude Code OpenClaw
Architecture 3 separate connection patterns Unified gateway with routing
Bridge Mode HTTP long-polling (2s interval) Centralized gateway routes requests
Remote Sessions WebSocket (receive) + HTTP (send) WebSocket through gateway
Routing Direct client-to-API connections All traffic through central gateway
Load Balancing API-level load balancing Gateway-level load balancing
Session Affinity Client polls same environment Gateway maintains session routing
Observability Client-side OpenTelemetry Gateway-level telemetry + client

Key Differences

Design Philosophy:

Claude Code Approach

  • Multiple specialized connection patterns for different use cases
  • Direct client-to-API connections (no central gateway)
  • Simpler architecture, lower operational overhead
  • Trade-off: Less centralized control

OpenClaw Approach

  • Unified gateway for all connection types
  • Central routing, load balancing, and session affinity
  • Gateway-level observability and control
  • Trade-off: More complex infrastructure

🏢 Enterprise Gateway Adaptations

Multi-Tenant Gateway Architecture:

For enterprise systems, you'll likely want an OpenClaw-style unified gateway:

1. Unified Gateway Benefits

  • Centralized authentication & authorization
  • Rate limiting per user/org
  • Request routing to agent instances
  • Load balancing across agent pools
  • Gateway-level caching
  • DDoS protection

2. Gateway Architecture

// Enterprise gateway stack ┌─────────────────────────────────────┐ │ Load Balancer (AWS ALB/NLB) │ └──────────────┬──────────────────────┘ │ ┌──────────────▼──────────────────────┐ │ API Gateway (Kong/Envoy/Custom) │ │ - Authentication (JWT) │ │ - Rate Limiting │ │ - Request Routing │ └──────────────┬──────────────────────┘ │ ┌──────┴──────┐ │ │ ┌───────▼─────┐ ┌────▼────────┐ │ Agent Pool │ │ Agent Pool │ │ (Org A) │ │ (Org B) │ │ K8s Pods │ │ K8s Pods │ └─────────────┘ └─────────────┘

3. Session Affinity

  • Route user to same agent instance for conversation continuity
  • Use sticky sessions or distributed cache (Redis)
  • Handle instance failures with session migration

4. WebSocket Gateway

// WebSocket gateway with session routing class WebSocketGateway { private sessions: Map = new Map(); async handleConnection(ws: WebSocket, userId: string) { // Route to existing agent or spawn new one const agent = await this.routeToAgent(userId); // Bidirectional message forwarding ws.on('message', (msg) => agent.send(msg)); agent.on('message', (msg) => ws.send(msg)); // Track for session affinity this.sessions.set(userId, agent); } async routeToAgent(userId: string): Promise { // Check if user has active agent if (this.sessions.has(userId)) { return this.sessions.get(userId)!; } // Spawn new agent from pool return await agentPool.getAgent(userId); } }

5. Gateway Observability

  • Gateway-level metrics (requests/sec, latency, errors)
  • Distributed tracing through gateway
  • Request/response logging for debugging
  • Real-time dashboards (Grafana, Datadog)
